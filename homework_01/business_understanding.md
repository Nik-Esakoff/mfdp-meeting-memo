# Домашнее задание №1  
## Бизнес-анализ проекта по автоматическому формированию мемо ВКС-встреч

## 1. Бизнес-анализ (Business Understanding)

### Цель и скоуп проекта

Цель проекта — разработать сервис, который автоматически обрабатывает запись ВКС-встречи и формирует по ней структурированное мемо.

Предполагаемый результат работы сервиса:

- текстовая расшифровка встречи;
- краткое содержание обсуждения;
- список принятых решений;
- список поручений;
- ответственные лица;
- сроки выполнения задач, если они были указаны во время встречи.

Основными пользователями сервиса будут сотрудники государственной компании, которые участвуют в рабочих ВКС-встречах и занимаются подготовкой, проверкой или использованием итоговых мемо.

Со стороны заказчика в проекте могут участвовать:

- руководитель подразделения или владелец бизнес-процесса;
- сотрудники, которые проводят ВКС-встречи;
- сотрудники, которые вручную формируют мемо;
- разработчик проекта;
- системные администраторы или инфраструктурные специалисты.

Для взаимодействия по проекту могут использоваться рабочие чаты, встречи и консультации с руководителем и пользователями будущего сервиса.

### Бизнес-цель проекта

Бизнес-цель проекта — упростить и ускорить процесс формирования мемо по итогам ВКС-встреч.

Сейчас сотруднику необходимо вручную прослушивать запись встречи, выделять основные темы, фиксировать решения, поручения, ответственных и сроки. Этот процесс занимает время и зависит от внимательности сотрудника.

Разрабатываемый сервис должен автоматически формировать черновик мемо, который пользователь сможет проверить и при необходимости отредактировать.

Ожидаемый эффект от внедрения:

- сокращение времени на подготовку мемо;
- снижение объёма ручной работы;
- уменьшение риска пропустить важное решение или поручение;
- приведение мемо к единому формату;
- ускорение распространения итогов встречи среди сотрудников.

### Существующие решения

На рынке существуют сервисы автоматической расшифровки и формирования итогов встреч, например Zoom AI Companion и другие облачные AI-сервисы.

Такие решения не полностью подходят для рассматриваемого проекта по следующим причинам:

- часть государственных и крупных компаний работает внутри собственного закрытого сетевого контура;
- записи встреч могут содержать конфиденциальную или служебную информацию;
- передача записей и транскриптов во внешние облачные сервисы может быть запрещена;
- внешние решения могут не учитывать внутреннюю терминологию, названия проектов, подразделений и фамилии сотрудников;
- заказчику может требоваться собственный формат мемо.

Поэтому необходимо локальное решение, которое может работать внутри инфраструктуры компании без передачи конфиденциальных данных во внешнюю сеть.

## 1.1 Текущая ситуация (Assessing Current Solution)

### Доступные ресурсы

Для реализации проекта имеется серверная инфраструктура компании.

Предполагается, что разработка и тестирование сервиса будут выполняться на имеющихся серверах без закупки дополнительного оборудования на первом этапе.

Перед началом разработки необходимо уточнить:

- характеристики доступных серверов;
- наличие GPU;
- объём оперативной и видеопамяти;
- возможность запуска контейнеров;
- доступный объём дискового пространства;
- ограничения внутренней сети.

### Данные

Основными данными для проекта являются:

- аудио- и видеозаписи ВКС-встреч;
- существующие мемо, подготовленные сотрудниками вручную;
- внутренние термины, названия проектов, подразделений и организаций;
- при необходимости — список участников встреч.

Данные хранятся на производственных серверах компании.

Для разработки потребуется согласованный доступ к части записей и готовых мемо. Данные должны быть обезличены либо использоваться только внутри закрытого контура компании.

Покупка или дополнительный сбор внешних данных на первом этапе не требуется. Для разработки предполагается использовать готовые предобученные модели распознавания речи и обработки текста.

### Эксперты проекта

В качестве экспертов предметной области могут выступать:

- руководитель автора проекта;
- автор проекта;
- сотрудники, которые регулярно участвуют во встречах и составляют мемо.

Эксперты смогут:

- определить требуемую структуру итогового мемо;
- выделить обязательные элементы документа;
- предоставить примеры корректных мемо;
- проверить качество результатов работы сервиса;
- определить, какие ошибки являются критичными.

### Риски проекта

#### Риск 1. Не уложиться в сроки

Проект включает несколько задач: распознавание речи, обработку текста, формирование мемо и развёртывание сервиса. Из-за слишком широкого объёма работ существует риск не завершить проект вовремя.

Меры по снижению риска:

- ограничить первую версию проекта минимальным функционалом;
- сначала реализовать цепочку «запись → расшифровка → мемо»;
- использовать готовые предобученные модели;
- не обучать большую модель с нуля;
- дополнительные функции реализовывать только после готовности базовой версии.

#### Риск 2. Получить сервис недостаточного качества

Качество сервиса может оказаться недостаточным из-за плохого звука, фонового шума, одновременной речи участников, ошибок распознавания фамилий и внутренних терминов, а также ошибок при формировании мемо.

Меры по снижению риска:

- протестировать несколько моделей распознавания речи;
- использовать внутренний словарь терминов и названий;
- проверять результат на реальных примерах встреч;
- предусмотреть обязательную ручную проверку итогового мемо;
- не использовать полностью автоматическую отправку мемо без подтверждения пользователя;
- собирать обратную связь и постепенно улучшать систему.

#### Риск 3. Ограниченный доступ к данным

Доступ к записям рабочих встреч может быть ограничен требованиями информационной безопасности.

Меры по снижению риска:

- выполнять обработку внутри закрытого контура;
- использовать обезличенные записи;
- не размещать реальные данные в открытом GitHub-репозитории;
- заранее согласовать правила хранения и обработки данных.

## 1.2 Решаемые задачи с точки зрения аналитики (Data Mining Goals)

Проект состоит из нескольких связанных задач машинного обучения:

1. распознавание речи;
2. формирование текстовой расшифровки;
3. выделение решений, поручений, ответственных лиц и сроков;
4. формирование структурированного мемо.

Для оценки результата недостаточно использовать только одну метрику, поскольку сервис должен одновременно правильно распознавать речь и формировать полезное итоговое мемо.

### Основная метрика качества

Основной технической метрикой предлагается использовать **F1-score для извлечения ключевых фактов из встречи**.

К ключевым фактам относятся:

- принятые решения;
- поручения;
- ответственные лица;
- сроки.

Для расчёта метрики результат сервиса сравнивается с мемо, которое было подготовлено и проверено экспертом.

F1-score учитывает одновременно:

- **Precision** — какая доля фактов, добавленных сервисом в мемо, действительно присутствовала во встрече;
- **Recall** — какую долю важных фактов из встречи сервис смог найти и добавить в мемо.

Использование только Precision недостаточно, поскольку сервис может пропускать важные поручения. Использование только Recall также недостаточно, поскольку сервис может добавлять факты, которых не было во встрече.

### Критерий успешности модели

Для первой версии проекта предлагаются следующие критерии:

- минимально допустимый F1-score — **0,70**;
- желаемый F1-score — **0,80 и выше**;
- Recall по решениям и поручениям — не ниже **0,75**;
- в итоговом мемо не должно быть критических вымышленных решений, ответственных лиц или сроков.

Указанные значения являются предварительными и могут быть скорректированы после анализа данных и консультации с экспертом.

### Дополнительные метрики

Для распознавания речи можно использовать метрику **WER — Word Error Rate**. Она показывает долю ошибок в распознанном тексте по сравнению с эталонной расшифровкой.

Предварительный критерий:

- допустимый WER — не более **30%**;
- желаемый WER — не более **20%**.

WER является вспомогательной метрикой. Даже при наличии ошибок в полной расшифровке сервис может корректно сформировать основные решения и поручения.

### Бизнес-метрика

Основной бизнес-метрикой будет сокращение времени на подготовку итогового мемо.

Для её оценки необходимо сравнить:

1. время, которое сотрудник тратит на полностью ручную подготовку мемо;
2. время, которое сотрудник тратит на проверку и редактирование мемо, созданного сервисом.

Критерий успешности проекта:

- минимальное сокращение времени — **30%**;
- желаемое сокращение времени — **50% и более**.

### Экспертная оценка

Поскольку качество итогового мемо невозможно полностью оценить автоматическими метриками, дополнительно будет использоваться экспертная оценка.

Руководитель или сотрудник, работающий с мемо, оценивает результат по шкале от 1 до 5 по следующим критериям:

- корректность;
- полнота;
- отсутствие вымышленных фактов;
- соответствие требуемой структуре;
- удобство дальнейшего использования;
- объём необходимых ручных исправлений.

Критерий успешности:

- средняя экспертная оценка не ниже **4 из 5**;
- мемо можно использовать после небольшой ручной проверки;
- в документе отсутствуют критические ошибки.

## 1.3 План проекта (Project Plan)

Проект будет выполняться итеративно и включать шесть этапов.

### Этап 1. Бизнес-анализ (Business Understanding)

Основные задачи:

- определить бизнес-цель проекта;
- определить основных пользователей;
- зафиксировать границы первой версии;
- определить доступные ресурсы;
- определить риски;
- выбрать бизнес- и ML-метрики.

Результат этапа:

- документ бизнес-анализа проекта.

### Этап 2. Анализ данных (Data Understanding)

Основные задачи:

- получить доступ к тестовым записям встреч;
- изучить качество и формат записей;
- оценить длительность встреч;
- проверить наличие готовых мемо;
- определить объём данных;
- изучить ограничения, связанные с конфиденциальностью;
- сформировать небольшой набор данных для тестирования.

Результат этапа:

- описание датасета;
- понимание качества и достаточности данных;
- набор примеров для разработки и оценки.

### Этап 3. Подготовка данных (Data Preparation)

Основные задачи:

- извлечь аудиодорожки из записей;
- привести аудио к единому формату;
- обезличить данные при необходимости;
- подготовить эталонные расшифровки для части записей;
- подготовить эталонные мемо;
- выделить решения, поручения, ответственных и сроки;
- разделить данные на выборку для разработки и итоговой оценки.

Результат этапа:

- подготовленный датасет;
- эталонные транскрипты и мемо;
- данные для расчёта метрик.

### Этап 4. Моделирование (Modeling)

Основные задачи:

- выбрать и протестировать готовую модель распознавания речи;
- получить базовую расшифровку;
- разработать формирование мемо;
- реализовать извлечение решений, поручений, ответственных и сроков;
- при необходимости добавить внутренний словарь или поиск по корпоративному контексту;
- сравнить несколько вариантов моделей и промптов.

Результат этапа:

- рабочий прототип пайплайна;
- несколько вариантов решения;
- результаты экспериментов.

### Этап 5. Оценка результата (Evaluation)

Основные задачи:

- рассчитать WER для распознавания речи;
- рассчитать Precision, Recall и F1-score для ключевых фактов;
- провести экспертную оценку мемо;
- сравнить время ручной подготовки и проверки автоматически созданного мемо;
- определить основные ошибки;
- решить, соответствует ли сервис критериям успешности.

Результат этапа:

- таблица метрик;
- экспертная оценка;
- вывод о качестве проекта;
- список направлений для улучшения.

### Этап 6. Внедрение (Deployment)

Основные задачи:

- подготовить простой интерфейс или API;
- развернуть сервис на внутреннем сервере;
- обеспечить обработку данных внутри закрытого контура;
- добавить журналирование ошибок;
- подготовить инструкцию для пользователя;
- провести тестирование на нескольких реальных встречах.

Результат этапа:

- демонстрационная версия сервиса;
- инструкция по запуску;
- результаты пилотного тестирования.

### Предварительная последовательность реализации

1. Подготовить несколько тестовых записей и мемо.
2. Реализовать распознавание речи.
3. Реализовать базовое формирование мемо.
4. Добавить выделение решений и поручений.
5. Рассчитать метрики.
6. Провести экспертную проверку.
7. Улучшить качество.
8. Развернуть демонстрационную версию на внутреннем сервере.
